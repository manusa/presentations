# DevTalks Romania 2026 — Slide Deck

Talk: **"Turning Your Java Project Into an AI-Ready Codebase"**
Event: DevTalks Romania, Java Stage, 03 June 2026, 14:35–15:20.

This directory hosts a **self-contained HTML/CSS/JS slide deck** that ships through the existing Gatsby static pipeline. The deck is **not** built with the React/Gatsby slide framework used by the older decks under `src/pages/presentations/*`.

---

## Why this deck is not Gatsby

Three constraints drove the call:

1. **Spectacular look with CSS animations / animated diagrams** — fastest medium for this is plain HTML+CSS+SVG in one file. PPTX/Google Slides lose 60–80% of the polish on conversion.
2. **Simple text editing** — each slide is a `<section>` block; changing a title is editing a string. Strictly simpler than the per-slide JSX components used by other decks in this repo.
3. **A one-week deadline** — no time to migrate or replicate the Gatsby framework for a brand-new design system.

The deck was generated by Claude Design (claude.ai/design) and exported as a handoff bundle. See the upstream conversation transcript in the original `devtalks-romania/chats/chat1.md` for design intent (cool electric purple/cyan base, orange personal-brand accent, Inter + JetBrains Mono, terminal motifs).

## What's in the bundle

When the bundle is dropped in here, the layout is:

```
2026-devtalks-romania/
  index.html           — single-page deck: <deck-stage> wrapping 12 <section> slides
  deck-stage.js        — vanilla-JS web component (keyboard nav, auto-scale to 1920×1080,
                          print-to-PDF, thumbnail rail, speaker notes)
  assets/              — images referenced by slides
  README.md            — this file
  AGENTS.md            — minimal orientation for future coding agents
  CLAUDE.md            — symlink → AGENTS.md
```

Slide model:

- Slides are direct children of `<deck-stage>`: `<section class="s-title" data-label="01 Title">…</section>`.
- 12 slide templates are provided: title, about, agenda, section divider, content, compare, quote, stat, code, diagram, full-bleed, closing.
- Theming is centralized in `:root` CSS custom properties at the top of `index.html` (`--bg`, `--accent`, `--accent-2`, `--pink`, `--purple`, type scale, padding tokens).
- Per-slide CSS is namespaced by class prefix (`.s-title { … }`, `.s-content { … }`, etc.).

Adding text to a slide = editing HTML inside the corresponding `<section>`. Adding a new slide = copy a section, change the class, add a CSS rule block.

## How it deploys

The repo has **two deploy paths**, both of which pick up this directory automatically because Gatsby copies `static/*` verbatim into `public/`:

| Channel | Workflow | Output URL |
|---|---|---|
| GitHub Pages | `.github/workflows/publish-gh-pages.yml` | `https://presentations.marcnuri.com/presentations/2026-devtalks-romania/` |
| NPM package | `.github/workflows/publish-npm.yml` | `npx mn-presentations` → local express server over `public/` → `http://localhost:8000/presentations/2026-devtalks-romania/` |

**No Gatsby integration is needed.** The deck is fully self-contained; no React, no SCSS pipeline, no JSX. Each push to `main` re-publishes both channels. The NPM publish requires a version bump in `package.json` to succeed (the GH Pages publish does not).

A link from the landing page (`src/components/landing-page/index.jsx`) is optional and can be added later if you want it listed alongside the Gatsby decks.

---

## Placement rationale

The directory sits at `static/presentations/2026-devtalks-romania/` (not `static/devtalks-romania/`) to match the URL shape of the Gatsby-generated decks, which all live under `/presentations/<slug>/`. This keeps the URL pattern consistent for anything (analytics, manual links, future indexing) that walks `presentations.marcnuri.com/presentations/*`.

Caveat: `README.md`, `AGENTS.md`, and `CLAUDE.md` will be publicly served at this URL because they live under `static/`. They are documentation, not secrets, so this is acceptable.

## Known tradeoffs / cleanups worth doing

- **Dev React/Babel includes**: `index.html` loads `react.development.js`, `react-dom.development.js`, and `@babel/standalone` from `unpkg`. These exist for the Claude Design *tweaks panel*, not the runtime deck. They can be stripped before shipping (or replaced with production builds if anything actually needs them — verify by removing and reloading).
- **CSS still embedded in one file**: ~990 lines of `<style>` at the top of `index.html`. A clean split into `styles/tokens.css` + `styles/shared.css` + `styles/s-*.css` is a 15–20 min mechanical refactor and pays off as the deck grows.
- **Per-slide split into separate HTML files** would need a tiny build step to concat back into a single `index.html` (because `<deck-stage>` operates on its direct DOM children). Not worth doing unless the single file becomes painful.

## Round-trip with Claude Design

Based on the handoff README and chat transcript, the export looks **one-way**: there is no documented import path that ingests modified files back into Claude Design. Working patterns:

- Iterate inside Claude Design until the master deck is close to what you want, hand off once, then continue here. Treat the bundle as the source of truth from that point on.
- For further visual exploration of *new* slide layouts, keep a parallel Claude Design session and copy the resulting `<section>` blocks into this deck manually.

This can be verified by checking the Claude Design UI directly — an import flow may exist that isn't reflected in the bundle.

## Authoring quick reference

| Task | Where | Effort |
|---|---|---|
| Change a title or body text | `<section>` HTML in `index.html` | seconds |
| Change theme colors / fonts | `:root` block in the `<style>` element | ~1 min |
| Add a status bar | duplicate the `.chrome-top` / `.chrome-bottom` pattern across sections, or move it into `deck-stage.js` for true persistence | 15–30 min |
| Add a new slide layout | copy a `<section>`, rename the class, add a `.s-mylayout { … }` rule block | 10–30 min |
| Add a CSS-animated diagram | inline SVG + `@keyframes` in the same `<style>` block | 30–60 min per diagram |

## Deploying a new version

1. Edit deck files in this directory.
2. Bump `package.json` `version` (root of repo) — required for the NPM publish workflow to publish a new tag. GH Pages publish does not need this.
3. Commit and push to `main`. Both workflows run automatically.
